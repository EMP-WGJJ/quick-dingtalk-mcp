# RFC: 内存与本地缓存控制策略

## 背景

当 MCP 处理 PDF 解析、邮件批量拉取等大数据量任务时，如果不加以限制，可能导致：

1. **内存溢出（OOM）**：大量数据一次性加载到内存中，导致 Node.js 进程崩溃。
2. **性能下降**：频繁的 GC 和大对象分配使响应时间显著增加。
3. **磁盘空间耗尽**：本地缓存文件无限增长，占满磁盘空间。

因此，需要分别针对 **内存** 和 **本地缓存** 设计合理的控制策略，保障 MCP 服务的稳定性和性能。

---

## 目标

- 防止单次任务或累积任务导致进程 OOM 崩溃
- 控制本地缓存文件的生命周期和总大小
- 提供可配置的阈值，适应不同部署环境
- 对现有工具调用透明，不破坏已有接口

---

## 设计方案

### 一、内存控制策略

#### 1.1 内存使用监控

- 使用 `process.memoryUsage()` 定期采样当前内存占用（RSS / heapUsed）。
- 设定 **警告阈值**（默认 70% 可用内存）和 **硬上限**（默认 85% 可用内存）。

#### 1.2 流式处理

- 对 PDF、邮件附件等大文件，采用 **流式读取（Stream）** 而非一次性加载到内存。
- 分片处理：将大数据拆分为固定大小的 chunk（默认 64KB），逐片处理后释放。

#### 1.3 内存压力响应

| 压力等级 | 条件 | 动作 |
|---------|------|------|
| 正常 | heapUsed < 70% | 正常执行 |
| 警告 | 70% ≤ heapUsed < 85% | 记录日志，降低并发度，延迟非关键任务 |
| 危险 | heapUsed ≥ 85% | 拒绝新任务，触发 GC，释放可清理缓存 |

#### 1.4 单任务内存限制

- 每个工具调用的数据处理设定最大内存配额（默认 50MB）。
- 超过配额时返回分页结果或错误提示，避免单任务耗尽全部内存。

---

### 二、本地缓存控制策略

#### 2.1 缓存目录管理

- 统一缓存目录：`~/.quick-dingtalk-mcp/cache/`
- 按工具/任务类型分子目录，例如：
  - `cache/pdf/`
  - `cache/mail/`
  - `cache/drive/`

#### 2.2 缓存淘汰策略

- **LRU（最近最少使用）**：优先淘汰最久未访问的缓存文件。
- **TTL（过期时间）**：缓存文件默认有效期为 24 小时，过期自动清理。
- **总大小限制**：缓存目录总大小上限默认 500MB，超出时按 LRU 淘汰。

#### 2.3 缓存写入控制

- 单文件大小上限：默认 100MB，超过该大小的文件不缓存，直接流式处理。
- 写入前检查剩余配额，空间不足时先触发淘汰再写入。

#### 2.4 定时清理

- 服务启动时执行一次全量清理（清除过期文件）。
- 运行期间每 30 分钟执行一次增量清理。

## 配置项

```javascript
// 默认配置示例
const DEFAULT_CACHE_CONFIG = {
  memory: {
    warningThreshold: 0.7,      // 内存警告阈值（占比）
    criticalThreshold: 0.85,    // 内存危险阈值（占比）
    maxTaskMemory: 50 * 1024 * 1024,  // 单任务最大内存 50MB
    chunkSize: 64 * 1024,       // 流式处理分片大小 64KB
    monitorInterval: 5000,      // 内存监控采样间隔 5s
  },
  disk: {
    cacheDir: '~/.quick-dingtalk-mcp/cache/',
    maxTotalSize: 500 * 1024 * 1024,  // 缓存总大小上限 500MB
    maxFileSize: 100 * 1024 * 1024,   // 单文件大小上限 100MB
    ttl: 24 * 60 * 60 * 1000,         // 缓存有效期 24h
    cleanupInterval: 30 * 60 * 1000,  // 清理间隔 30min
    evictionPolicy: 'lru',            // 淘汰策略
  }
};
```

## 影响范围

以下工具模块在处理大数据时将受益于此策略：

| 模块 | 场景 |
|------|------|
| `mail/message` | 批量拉取邮件及附件 |
| `drive/file` | 大文件下载/上传 |
| `doc/document` | 文档内容解析 |
| `sheet/data` | 大量表格数据读取 |
| `aitable/record` | 批量记录查询 |
| `aitable/data_io` | 数据导入导出 |

## 风险与注意事项

1. 流式处理改造需保持向后兼容，不破坏已有工具的返回格式。
2. 缓存清理需加文件锁，避免并发读写冲突。
3. 配置项需支持环境变量覆盖，方便不同环境部署。
